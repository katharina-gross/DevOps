## **Задание для стажера DevOps (Docker + gitlab-ci, Kubernetes + Helm)**  
**Цель:** Продемонстрировать навыки создания безопасного образа приложения; настройки автоматизации сборки, тестирования и проверок на уязвимости; развертывания в Kubernetes.
В качестве прикладного сервиса используется Spring Boot приложение https://github.com/EugeneKozyrev/book-api, сборка должна осущесвлятся с помощью maven, образ должен формироваться используя команду `docker build`. Важной скрыть из исходников все чуствительные данные. Код предоставить для проверки опубликовав его в публичном репозитории.

DoD не фиксирован. Даже если какая-то часть задания не выполнена, результат будет оценен и по решению проверяющего зачтен.

### **Базовая часть** 

#### **1. Написать Dockerfile, который соответствует практикам безопасной разработки и сборки ПО**
- Использование многоступенчатой сборки (использовать разные образы для сборки и запуска);
- Использование минималистичного базового образа;
- Запуск процесса не от имени root (в Dockerfile присутствует конструкция `USER`);
- Отсутствие лишних файлов в образе (в образе присутствуют только необходимые для работы файлы);
- `docker build` проходит успешно, `docker run` проходит успешно, есть доступ к API прикладного сервиса.

#### **2. Установка и настройка локального кластера**  
- Установить **`kind`** (Kubernetes in Docker) или **`minikube`**.  
- Запустить кластер и проверить подключение через **`kubectl`**.  
- Убедиться, что `kubectl get nodes` показывает рабочую ноду. 
- Выполнить `kubectl get ns`, почитать о концепции namespace в k8s
- Выполнить `kubectl get po -n kube-system`, ознакомиться с контейнерами запущенными в неймспейсе (что это за контенеры и зачем они нужны).

#### **3. Запуск пода с прикладным сервисом** 
- Создать **Pod** (`kubectl run`) взяв за основу образ собранный на этапе 1
- Проверить логи (`kubectl logs <pod> -c <container>`).
- Получить доступ к api через `kubectl port-forward`

#### **4. Переход на ReplicaSet и Deployment**  
- Заменить **Pod** на **Deployment** (указать `replicas: 2`) (https://kubernetes.io/docs/concepts/workloads/controllers/deployment/)
- Проверить, что при удалении пода он восстанавливается. 

#### **5. Настройка доступа через Service**  
- Создать **Service** типа **ClusterIP** для доступа к поду внутри кластера.   
- Получить доступ к api через **Service**.

### **Продвинутая часть часть** 

#### **6. Подготовить Helm-чарт или YAML манифесты для развертывания в K8s сервиса собранного ранее, соблюдая**
- Resource Limits: Установка лимитов CPU и памяти;
- Security Context: Настройка ограничения прав;
- Secrets: Не хранить ключи в манифестах;
- Network Policies: Ограничить входящий трафик только на порт приложения;
- Liveness/Readiness Probes: Правильная настройка для стабильности работы и запуска.

#### **7. Настроить автоматизацию (перечень команд, скрипт, gitlab-ci), которая**
- Собирает проект через maven;
- Запускает статический анализ кода (инструмент можно взять любой);
- Собирает Docker-образ;
- Сканирует собранный образ на уязвимости (инструмент можно взять любой);
- Пушит образ в реестр (можно опустить если среда развертывания совпадает со средой сборки);
- Формирует отчет о найденных уязвимостях (формат отчета произвольный).

#### **8. Задокументировать**
- Почему выбраны те или иные технологии и инструменты;
- Как запустить автоматизированую сборку и проверки;
- Как проверить наличие уязвимостей.
